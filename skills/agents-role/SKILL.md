# Agents角色设计技能

## 概述

本技能用于设计AI Agent的完整角色定义，生成4个标准文档：
- **IDENTITY.md** - 身份设定（Name、Role、Tagline、Description、Expertise）
- **SOUL.md** - 核心人格（Personality、Values、Communication Style、Emotional Response、Humor、Boundaries、Tone）
- **USER.md** - 用户画像（背景、偏好、业务上下文）
- **AGENTS.md** - 操作规则（核心原则、交互方式、输出格式、团队协作）

## 触发条件

以下情况应触发此技能：
- 用户说"设计一个XX角色"
- 用户说"创建agents文档"
- 用户说"定义AI人格"
- 用户说"帮我写SOUL.md"
- 用户说"创建角色定义"
- 用户要求生成4个文档文件
- 用户描述了一个角色（如"帮我设计一个老派Unix工程师，毒舌但其实是为你好"）

## 核心原则：现场推断角色特征

**本技能不预设任何角色模板**。根据用户的描述，现场推断这个角色应该是什么样的真人/AI，然后生成符合该气质的内容。

### ⚠️ 关键原则：精简提取

**不要把用户输入的全部内容都采纳**，只提取最核心的特征：
- 用户可能会给一大段描述 → 提取 2-3 个关键特征即可
- 避免过度设计 → 角色越简单越有记忆点
- 示例：
  - 用户说："帮我设计一个老派Unix工程师，命令行不离手，毒舌但其实是为你好，喜欢用各种终端工具，偶尔也会用用vim，虽然嘴上说图形界面是小朋友用的但其实有时候也会用用IDE"
  - → 精简为：**老派 + 毒舌 + 命令行**（其他细节可以现场补充）

### 步骤1: 理解用户想要什么样的角色

用户可能提供：
- **直接类型**："帮我设计一个架构师" → 询问更多细节或根据"架构师"推断特征
- **具体描述**："帮我设计一个老派后端工程师，命令行不离手，毒舌但其实是为你好" → 根据描述现场推断
- **感觉描述**："帮我设计一个温暖的、像大姐姐一样的AI" → 根据感觉生成

**精简提取**：只抓最核心的 2-3 个特征，不要试图覆盖用户说的每一点。

如果信息不足，询问用户：
- "这个角色的核心特点是什么？"
- "你希望他/她/它说话是什么风格？"
- "这个角色的主要用途是什么？"

### 步骤2: 现场推断角色特征

根据用户的描述，推断以下内容：

| 推断维度 | 问题 |
|---------|------|
| 名字风格 | 这个角色应该叫什么名字？（如"毒舌老张"、"萌妹小爱"） |
| 气质风格 | 给人什么感觉？（如"毒舌但温柔"、"高冷但可靠"） |
| 说话方式 | 怎么说话？（如"喜欢用命令句"、"爱用颜文字"） |
| 核心信念 | 这个角色相信什么？（如"命令行是最好的老师"、"代码即艺术"） |
| 边界限制 | 这个角色不会做什么？（如"绝不说你很笨"、"绝不原谅不写测试的代码"） |
| 交互方式 | 会怎么回应用户？（如"毒舌提醒"、"温柔鼓励"、"直接给答案"） |

### 步骤3: 生成符合角色的内容

**根据步骤2的推断，生成4个文档：**

- **IDENTITY**: 根据气质风格生成名字、Vibe、Emoji
- **SOUL**: 根据核心信念生成价值观、沟通风格、边界
- **AGENTS**: 根据交互方式生成具体回应规则
- **USER**: 询问用户信息

### 示例

**用户输入**: "帮我设计一个老派Unix工程师，命令行不离手，毒舌但其实是为你好"

**现场推断**:
- Name: 老张 / 终端大师 / Unix老人
- Vibe: 毒舌但可靠、命令行思维、傲娇
- Emoji: 💻 或 👴
- 说话方式: 命令句、术语、毒舌提醒
- 核心信念: "一切皆文件"、"不会命令行就是菜"、"图形界面是小朋友用的"
- 边界: 绝不说"这很简单"、绝不容忍"rm -rf /"的玩笑
- 交互方式: 毒舌提醒但会给正确做法

**生成内容示例**:
```markdown
# IDENTITY.md
Name: 老张
Vibe: 毒舌但可靠、命令行思维、傲娇
Emoji: 💻

# SOUL.md
核心真理:
- 一切皆文件，不会命令行就是菜
- 图形界面是小朋友用的
- 真正的工程师用终端
- 毒舌是因为在乎，不然才懒得说

边界:
- 绝不说"这很简单"
- 绝不容忍"rm -rf /"的玩笑
- 绝不对命令行错误视而不见
```

## 输入信息

需要从用户处获取：
1. **角色描述** - 用户想要什么类型的角色，或者具体的角色特征描述
2. **输出目录** - 文件输出位置
3. **用户信息**（USER.md）- ⚠️ **每个必填字段都必须明确询问用户，不能假设**：
   | 字段 | 是否必填 | 示例 |
   |------|---------|------|
   | 姓名 | ✅ 必填 | icer |
   | 称呼 | ✅ 必填 | bro、哥、大佬 |
   | 时区 | ✅ 必填 | Asia/Shanghai |
   | 备注 | ✅ 必填 | 资深高级开发 |
   | 代称 | ❌ 可选 | 他/她/他们 |

## 模板目录

模板位于 `{skill目录}/templates/`，使用绝对路径读取：

```
{skill目录}/templates/
├── identity.md  - IDENTITY.md 模板
├── soul.md      - SOUL.md 模板
├── user.md     - USER.md 模板
└── agents.md   - AGENTS.md 模板
```

> 例如：`/Users/ice/Go/src/surrogate/skills/agents-role/templates/identity.md`

## 使用方式

1. **读取模板** - 读取 `templates/` 目录下的模板文件
2. **收集需求** - 向用户确认角色类型、定位、目标用户
3. **填充内容** - 根据角色特点填充各文档的详细内容
4. **生成文件** - 在用户指定目录创建4个文档

## 各文档填充指南

### IDENTITY.md 填充内容

根据推断的角色特征填充：

| 字段 | 说明 | 填充依据 |
|------|------|---------|
| Name | 角色名称 | 根据气质风格定，如"老张"、"小爱"、"大师" |
| Creature | 是什么 | AI？机器人？灵宠？根据角色设定 |
| Vibe | 给人什么感觉 | 从描述中推断，如"毒舌但温柔"、"高冷可靠" |
| Emoji | 签名表情 | 符合角色气质 |
| Notes | 备注 | 角色的简短描述 |

### SOUL.md 填充内容

根据推断的角色特征填充：

| 字段 | 说明 | 填充依据 |
|------|------|---------|
| 核心真理 | 角色相信的信念 | 根据角色使命推断，如"命令行是最好的老师" |
| 边界 | 不会做什么 | 根据角色性格推断，如"绝不说你很笨" |
| 气质 | 整体感觉 | 根据Vibe延伸 |

### USER.md 填充内容

USER.md 需要从用户处收集以下信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 姓名 | 用户的名字 | icer |
| 怎么称呼你 | 用户喜欢的称呼 | bro |
| 代称 | 他/她/他们（可选） | pro |
| 时区 | 如 Asia/Shanghai | Asia/Shanghai |
| 备注 | 用户角色/项目背景 | 项目负责人 |

其他可选字段：
| 字段 | 说明 | 示例 |
|------|------|------|
| 背景 | 用户职业、团队规模、技术栈 | 生鲜配送SaaS ERP创始人 |
| 偏好 | 语言、沟通风格、决策方式 | 中文、直接高效、务实导向 |
| 业务上下文 | 产品、目标客户、核心业务 | B端SaaS生鲜配送ERP |

### AGENTS.md 填充内容

AGENTS.md 由三部分组成：

#### 1. 记忆系统（默认保留）
读取 `templates/agents.md`，直接使用其中的"记忆系统"部分（每次会话、memory/YYYY-MM-DD.md、长期记忆 MEMORY.md、写下来规则），不需要修改。

#### 2. 安全规则（默认保留）
直接使用模板中的"安全"和"外部 vs 内部"规则，不需要修改。

#### 3. 角色自定义部分
根据推断的角色特征填充：

| 字段 | 说明 | 填充依据 |
|------|------|---------|
| 核心原则 | 最重要的行为准则 | 根据角色使命推断 |
| 交互方式 | 具体场景的回应方式 | 根据角色性格推断，如毒舌→提醒型，温柔→鼓励型 |
| 绝不会做的事 | 底线 | 根据角色边界推断 |

**生成顺序**：记忆系统 → 角色自定义部分 → 安全规则 → 外部 vs 内部

## 执行步骤

### ⚠️ 严重警告

1. **必须先备份目录才能执行后续操作**
2. **所有生成或替换的内容必须先由用户确认后才能创建/修改**（快速模式除外）

### 两种模式

| 模式 | 说明 |
|------|------|
| 确认模式（默认） | 生成前展示预览，等用户确认"可以生成" |
| 快速模式 | 用户明确说"直接生成"时，跳过确认步骤 |

### 执行流程

1. **理解角色需求 + 收集用户信息** - 同时进行：
   - 从用户描述中理解想要什么样的角色
   - **⚠️ 必须逐个确认 USER.md 的每个字段都有值**：
     - 姓名：用户的真实姓名（必填）
     - 称呼：如 bro、哥、大佬（必填）
     - 时区：如 Asia/Shanghai（必填）
     - 备注：如 资深高级开发、项目负责人（必填）
     - 代称：（可选）
   - **不要假设任何字段的值，全部问用户确认**
   - 如果描述具体 → 直接推断角色特征
   - 如果描述模糊 → 询问补充信息
2. **现场推断角色特征** - 根据描述推断名字、气质、说话方式、核心信念、边界、交互方式
3. **校验输出目录** - 必须从用户获取明确的输出目录路径
4. **备份现有目录** - 直接执行备份，无需判断是否存在
   - 命令：`cp -r {输出目录} {输出目录}_back_{日期时间}`
   - 备份完成后才能继续
   - 告知用户备份位置
5. **读取模板** - 读取 templates/ 下的4个模板（使用绝对路径）
6. **读取目标文件** - **⚠️ 必须先读取现有文件才能写入**
   - 使用 Read 工具读取目标目录下的现有文件
   - 目的：避免 Write 报错
7. **填充内容** - 根据推断的角色特征填充详细内容
8. **用户确认 或 直接生成**
   - **确认模式**：展示4个文档预览，等待用户确认"可以生成"
     - ⚠️ **重点检查 USER.md**：确认姓名、称呼、时区、备注 都有正确的值
     - 如果发现空值，立即补充或再次询问用户
   - **快速模式**：用户说"直接生成"时，跳过确认步骤
9. **生成文档** - 创建/修改用户指定目录下的4个文档
10. **返回结果** - 告诉用户文件生成的位置和备份位置

## 注意事项

- **不预设角色模板** - 根据用户描述现场推断，不套用任何预设
- **生成的4个文档应该是一个整体，相互呼应** - IDENTITY定义"是谁"，SOUL定义"怎么表现"，AGENTS定义"怎么做"
- **USER.md的内容从与用户的对话中推断或由用户提供**
- **参考 templates/ 目录的模板结构，但填充具体内容**
- **所有文档使用中文输出**
- **如果用户描述不够具体，一定要询问补充信息**
- **⚠️ 必须先读后写** - 在写入任何文件前，必须先用 Read 工具读取现有文件，否则 Write 会报错
- **备份总是执行** - 不判断目录是否存在，直接执行 cp 命令（空目录也会被复制，无副作用）
